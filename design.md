# System Design Foundation

## 1. Architecture

This system uses a split frontend/backend architecture:

- Frontend (React + Vite):
  - Public views: gate, login, register
  - Protected dashboards by role
- Backend (Express + Prisma):
  - API endpoints, auth, role control
  - Rule-aware attendance calculation layer (next phase)
- PostgreSQL:
  - Source of truth for users, shifts, rules, attendance, roster, leave
- Redis:
  - Cache/session support for token and high-frequency reads (next phase)

## 2. Role Model

Roles:

- `ADMIN`: full configuration and governance
- `HR`: employee lifecycle, roster, approvals
- `EMPLOYEE`: attendance and leave operations
- `SECURITY`: employee dashboard access with security role identity

Role-based routing implemented on frontend and backend middleware.

## 3. Employee Code Strategy

Employee code format is generated by category prefix + 4-digit running number:

- `ADMIN` -> `AD-0001`
- `HR` -> `HR-0001`
- `SECURITY` -> `SG-0001`
- `EMPLOYEE + WORKER` -> `GW-0001`
- `EMPLOYEE + STAFF` -> `ST-0001`

`employeeCategory` is added to support Worker vs Staff split while keeping `role=EMPLOYEE`.

## 4. Data Model Highlights

Key entities:

- `User`, `Department`, `Shift`
- `AttendanceRule` for global policy configuration
- `Registration` for approval workflow
- `Roster` for shift assignment per date
- `Attendance` for clock and computed status
- `LeaveApplication` for leave lifecycle

Attendance and roster include indexes for operational query patterns.

## 5. Auth & Security Flow (Foundation)

- Access token is read from `Authorization: Bearer <token>`.
- Backend middleware verifies JWT and attaches user to `req.user`.
- Authorization middleware enforces allowed roles.
- Frontend Axios interceptor injects access token and handles `401` with redirect to `/login`.

Planned enhancements:

- Refresh-token rotation endpoint
- HttpOnly cookie strategy for refresh token
- Rate limiting and IP throttling

## 6. Attendance Rule Engine (Planned)

The schema already supports required fields for rule-driven calculations:

- Late threshold and excessive late thresholds
- Early exit flag windows
- Ramadan date windows and iftar/sehri conflict windows
- OT policy (`AFTER_8_HOURS`) and security OT exclusion

Next implementation phase should encapsulate calculations under `backend/src/lib/rules`.

## 7. Deployment Direction

- Containerize backend and frontend separately
- Apply Prisma migrations in CI/CD pipeline
- Store secrets via environment variables only
- Use managed PostgreSQL/Redis in production
